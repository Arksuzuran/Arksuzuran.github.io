Thinking

Thinking 1.1

​ 请阅读附录中的编译链接详解，尝试分别使用实验环境中的原生 x86 工具 链（gcc、ld、readelf、objdump 等）和 MIPS 交叉编译工具链（带有 mips-linux-gnu前缀），重复其中的编译和解析过程，观察相应的结果。

​ 只进行预处理，通过diff命令查看预处理结果的不同，主要体现在三个方面：

​ 1.头文件存放的位置不同。x86架构所需要的头文件主要在/usr/include/目录下，mips架构所需要的头文件主要在/usr/mips-linux-gnu/include/目录下。

​ 2.mips架构下部分typedef语句前带有__extension__，而x86-64架构下则没有。该关键词作用是使得gcc不再对标准C语言的扩展功能进行警告。

​ 3.mips架构下使用了诸如__uint64_t，__int64_t等宏定义，而x86-64架构下是清一色的long int，unsigned long int 等。这是为了使得所需的整数类型在32位的mips下依旧为固定的64位长度（C的标准只规定特定数据类型需要实现的最小长度，特定类型的具体长度取决于编译器实现。为了增强程序的可移植性，C99标准增加了对固定长度的整数类型的支持）。

​ 只进行编译而不链接，对产生的o目标文件使用objdump进行反汇编，再使用diff命令比较反汇编结果的不同，主要体现在三个方面（图为text节的不同之处）：

​ 1.x86-64使用变长汇编指令，而mips指令集固定为4字节。

​ 2.x86-64的地址为64位，而mips架构的地址为32位。例如图中main函数在x86-64下的地址为00000000000000000，而在mips下地址为00000000。

​ 3.section的名称、数量、顺序不同。

Thinking 1.3

在理论课上我们了解到，MIPS 体系结构上电时，启动入口地址为 0xBFC00000 （其实启动入口地址是根据具体型号而定的，由硬件逻辑确定，也有可能不是这个地址，但一定是一个确定的地址），但实验操作系统的内核入口并没有放在上电启动地址，而是按照内存布局图放置。思考为什么这样放置内核还能保证内核入口被正确跳转到？

​ 我们的实验使用GXemul仿真器，通过直接运行ELF格式的可执行文件mos来实现MOS操作系统的运行。启动流程的两个阶段为：加载内核到内存，跳转到内核的入口。

  • 加载内核到内存

    该阶段已经由GXemul仿真器帮助我们完成。其中，内核被放置的位置在链接产生mos可执行文件时就已经确定（由Linker Script文件kernel.lds指定）。

  • 跳转到内核的入口

    由内核本身完成。在链接产生mos可执行文件时，根据kernel.lds，**ENTRY(_start)已经将内核的入口被指定为_start 。而内核向C语言部分的跳转则由_start 完成**：设置栈指针，并跳转到C函数mips_init()。

难点分析

1.ELF文件的功能与格式

​ ELF（Executable and Linkable Format）是Linux系统下，可重定位文件（.o目标文件，包含二进制代码和数据，可以与其他目标文件合并创建一个可执行文件）、可执行文件（包含二进制代码和数据，可直接被加载器加载执行）以及共享对象（库）文件的常用文件格式。

图片来自 ELF文件详解—初步认识_code&poetry的博客-CSDN博客

​ ELF文件有四个部分：

  • ELF头。包括了程序的基本信息：体系结构、操作系统、节头表和段头表相对文件的偏移量等。ELF头实际上是一个存储了ELF文件信息的结构体。

  • 段头表（program header table）。段头表主要包含程序中各个段（segment）的信息， 段的信息需要在运行时刻使用。段头表可以看做是一个结构体数组，每个元素都记录了该段数据载入内存时的目标位置等，记录了用于指导应用程序加载的各类信息。

  • 节头表（section header table），主要包含程序中各个节（section）的信息，节的信息需要在程序编译和链接的时候使用。节头表可以看做是一个结构体数组，每个元素记录了该节程序的代码段、数据段等各个段的内容，主要是链接器在链接的过程中需要使用。

  • 节或段。从汇编器和链接器的视角看，上图中蓝色部分会被当做节的集合进行处理；从加载器（执行ELF文件时）的角度来看，这部分会被当做段的集合进行加载。

    section和segment的区别：

    我们在汇编程序中用.section声明的Section会成为目标文件中的Section，此外汇编器还会自动添加一些Section（比如符号表）。

    Segment是指在程序运行时加载到内存的具有相同属性的区域，由一个或多个Section组成，比如有两个Section都要求加载到内存后可读可写，就属于同一个Segment。有些Section只对汇编器和链接器有意义，在运行时用不到，也不需要加载到内存，那么就不属于任何Segment。
    https://blog.csdn.net/daide2012/article/details/73065204

    三个重要的节：

    • .text：保存可执行文件的操作指令。
    • .data：保存已经初始化的全局变量和静态变量。
    • .bss：保存未初始化的全局变量和静态变量。

​ 实验中，要求以节的形式对ELF文件进行解析。在我们自写的readelf中，首先验证文件是否为有效的ELF（检验ELF头中存储的魔数），然后通过解读ELF头获取节头表的地址。通过解读节头表数组的各个元素，就能够获取各个节的基本信息。Exercise1.1的解题思路如下图所示：

2.MIPS内存布局

​ 我们的操作系统本身也是程序，因此使用虚拟地址。MIPS架构下的4G虚拟地址空间划分如下：

​ 当操作系统启动时，MMU还没有被初始化，但cache已经被bootloader初始化，因此内核只能被放置在kseg1或者kseg0。而kseg1又用于访问外设，因此内核被加载至kseg0处。

​ Linker Script 中记录了各个节应该如何映射到段，以及各个段应该被加载到的位置。通过设置Linker Script，即可让链接器把内核调整到指定的地址上。

​ 在start.S的最后一步，我们跳转到了内核的C语言部分（mips_init函数）。由于此处我们的内核本质上是C正在运行的程序，因此必须将栈指针初始化。C语言的运行需要使用栈，其作用大致为：保存上下文，传递参数。每一个未运行完的C函数都有其对应的栈帧，用以记录其运行状态，这样才能实现函数的调用。

实验体会

​ 刚刚做完Lab1时，感觉题目之间联系不大，有些割裂感。实验题目本身难度并不大，我在没有理解透彻指导书的情况下稀里糊涂地完成了Exercise。而当我回头完成了思考题和实验报告时，才发现其中紧密的逻辑关系。

​ 我们的实验使用GXemul仿真器，它能够直接读取ELF格式的可执行文件来进行模拟，而ELF文件则是在Makefile的规则下一步步搭建起来的（每一个带有Makefile的目录，其下的文件都参与了内核的构成）。

​ 既然内核是ELF格式的文件，那么我们自然要对ELF文件进行学习，了解到其解析方法，以及节和段的用途和区别。

​ 内核毕竟也是程序，需要使用虚拟内存。要用虚拟内存，那就不得不涉及到加载位置的问题。在链接时，Linker Script控制节的加载地址，而加载内存时的主体——段，又由节组成，因此设置好了节，就等同于将内核安置好了。这正好用到了上文中ELF文件的知识。

​ 最后，当内核被加载到内存中时，再通过汇编语言入口，设置好栈指针，为C程序的运行做好一切准备，再跳转到C语言部分，我们的内核就启动完成了。
