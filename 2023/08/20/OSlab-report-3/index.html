Thinking 3.1

请结合 MOS 中的页目录自映射应用解释代码中 e->env_pgdir[PDX(UVPT)] = PADDR(e->env_pgdir) | PTE_V 的含义。

​ 含义是根据线性映射规则，设置好用户进程的页目录自映射：即页目录中的第PDX(UVPT)项记录页目录自己的起始物理地址PADDR(e->env_pgdir) ，并赋予该项只读权限。

​ 对于虚拟地址空间内的任一个虚拟页，根据线性映射规则，都对应一个唯一的二级页表项。在页目录的某一项内写入管理该虚拟页的二级页表，然后再在这个二级页表的某一项内写入我们希望该虚拟页对应的实际物理地址和权限位，就建立起了一组映射关系。整个页表区UVPT本身占了4MB，正好需要一个二级页表来管理，而这个二级页表恰好就是页目录自身，所有我们才需要在页目录里登记页目录自己的物理地址。页目录自映射就是在页目录的特定项中填入页目录起始物理地址 | perm。

​ 页目录自身的进程页表区域的起始地址是UVPT，故页目录是第PDX(YVPT)个二级页表，映射到页目录自身的页目录项是第PDX(YVPT)项。而e->env_pgdir是页目录在kseg0里的虚拟地址，故直接令 e->env_pgdir[PDX(UVPT)] = PADDR(e->env_pgdir) | PTE_V即可设置好页目录自映射。

​ 如此一来，进程便可以在用户态下查询自己的页表。当PTE_D为0时触发TLB_MOD异常，在处理函数do_tlb_mod里，我们将返回地址设置为curenv->env_user_tlb_mod_entry，从而实现在用户态下处理该异常。判断是否需要写时复制时，就需要访问进程自己页表的perm位。

​ 建立起这样的自映射有什么用呢？应当是在用户需要读取虚拟地址空间UVPT到ULIM之间的内容时，供TLB miss时重填用（Lab2.8-2.10）。

Thinking 3.7

阅读相关代码，思考操作系统是怎么根据时钟中断切换进程的。

​ 当时钟中断发生时，硬件设置SR寄存器的二重栈，并设置Cause寄存器，以记录异常原因。随后CPU从0x80000000取指，开始执行通用异常分发程序。

​ 在异常分发程序中，先保存上下文到内核异常栈，然后根据Cause寄存器的值，异常分发程序会跳转到handle_int函数。

​ 在handle_int函数中，进行掩码计算：如果Cause寄存器和SR寄存器第14位（IM4）的值都为1，那么就会触发时钟中断，即执行timer_irq，响应时钟中断并不返回地（因为调度过后PC应该在新进程的虚拟地址空间上）跳转到进程调度函数schedule，其中参数yield为0。

​ 在调度函数schedule中，执行时间片轮转算法：根据是否调度过进程、当前进程是否就绪、剩余时间片数量、参数yield来决定是否切换进程。如果需要切换进程，则判断当前进程是否插入调度链表尾部。最后，调用env_run，将当前进程的现场保存到进程控制块中，然后切换到调度链表头部结点的进程。如果不需要切换进程，减少当前进程剩余的时间片数量。

一句话重点：env_sched_list中元素需要保持e->env_status==ENV_RUNNABLE.

​ LIST_ENTRY(Env) env_link与TAILQ_ENTRY(Env) env_sched_link用来连接队列空闲队列env_free_list和调度队列env_sched_list。

（1）空闲队列：

​ 当有进程的e->env_status改为或失去ENV_RUNNABLE时需要调用：env_alloc(struct Env **new, u_int parent_id)，env_free(struct Env *e)

（3）注意点：

​ 本实验中TAILQ与LIST的区别有以下两点：

  • TAILQ有尾指针，而LIST没有。
  • TAILQ额外允许尾部插入操作TAILQ_INSERT_TAIL(&env_sched_list, env, env_sched_link)，而LIST只允许头部插入操作。

​ 需要格外注意的是，当取链表的头元素时，应当先判断是否为空:TAILQ_EMPTY(&enc_sched_list)，如果为空则一般会报错或者panic。例如schedule(int yield)产生panic，env_alloc(struct Env **new, u_int parent_id)报错-E_NO_FREE_LIST。

1.4 env_tf（

​ 当进程的执行被迫中断（执行进程调度，或者执行syscall_*陷入内核时）时用以保存进程的上下文（即部分寄存器的状态），以备后续执行RESTORE_SOME时恢复。

​ regs数组用来保存32个通用寄存器。其中常用的几个是：

  • regs[2]: v0寄存器。存储函数的返回结果，手动设置以人工指定函数返回值
  • regs[4-7]: a0-a3寄存器。存储函数调用时的前四个参数。
  • regs[29]: sp寄存器，栈指针。栈分为用户栈（栈底）和内核栈。

​ 其余寄存器的作用如下：

  • cp0_status：即SR寄存器。是状态寄存器，包括中断引脚使能，其他 CPU 模式等 位域。

​ 15-8 位为中断屏蔽位，每一位代表一个不同的中断活动，其中 15-10 位使能外部中断源，9-8 位 是 Cause 寄存器软件可写的中断位。IM(x)对应第x号中断，为1时代表开启对应中断。其中4号中断是时钟中断。

中断号和异常号是不一样的概念，我们实验的异常包括中断，中断本身是 0 号异常，时钟中断又是4号中断。

​ 低6位是一个二重栈结构，其中KU位代表是否处于用户模式下，IE表示中断是否开启。KUc和IEc代表当前CPU的实际运行状态。

当异常发生时，CPU会自动将栈左移两位，同时KUc和IEc置0，表示CPU进入内核态并禁止中断。

当执行ret_from_exception即将返回用户态时，最后的rfe指令会将栈右移两位，即KUc和IEc恢复为KUp和IEp状态。

​ 每一次时钟中断完成后，进程被调度时，都会执行rfe指令。

  • hi：即 EntryHi，保留TLB的Key。

  • lo：即 EntryLo，保留TLB的Data。

  • cp0_badvaddr：即BadVaddr，保存引发地址异常的虚拟地址。

  • cp0_cause：即Cause寄存器，15-8 位保存着哪一些中断发生了，其中 15-10 位来自硬件，9-8 位可以由软件 写入，当 SR 寄存器中相同位允许中断（为 1）时，Cause 寄存器这一位活动就会导致中断。6-2 位（ExcCode），记录发生了什么异常。

  • cp0_epc：即EPC寄存器，存储进程恢复运行时 PC 应恢复到的位置，CPU 将自动 从 PC 所指的位置开始执行二进制码。

2.进程的创建过程

2.1 段地址映射函数map_segment

​ 函数void map_segment(Pde* pgdir, u_long pa, u_long va, u_long size)的作用是，在页目录pgdir确定的虚拟地址空间内，将虚拟地址段[va,va+size) 按页映射到物理地址段 [pa,pa+size)，并将相关页表项的权限设置为 perm。

内核态函数，pgdir要求为kseg0中的地址；va是kuseg中的虚拟地址；pa, va, size必须按页(BY2PG)对齐。

2.2 进程地址空间初始化函数env_setup_vm

​ 函数static int env_setup_vm(struct Env *e)的作用是初始化进程的虚拟地址空间，设置好进程目前地址空间内的三个段：

  • UENVS~UPAGES-1：映射到envs所在物理页，所有进程共享的只读空间。
  • UPAGES~UVPT-1：映射到pages所在的物理页，所有进程共享的只读空间。
  • UVPT~ULIM-1：用户进程的页表区。

​ 这三个段的大小都是4M且对齐，这意味着他们都被一个二级页表所管理。在页目录的三个相应表项内填入这三个二级页表的物理地址|perm，即可完成映射。因此我们的工作分为三步：

  • 分配一个物理页作为页目录
  • 设置好映射envs与pages
  • 设置好自映射

​ 考虑到所有用户进程的UENVS~UVPT-1都映射了envs和pages，因此我们可以先构造一个填好了这两项的页目录模板base_pgdir。这样做的好处主要是可以复用管理envs和pages的二级页表，从而节约物理空间。

​ base_pgdir是kseg0中的虚拟地址。

复习：

Pde, Pte: u_long, 即unsigned long

PDX(va)：求va对应的页目录项下标。va可以是4G空间内的任意虚拟地址。

PTX(va)：求va对应的二级页表下标。va可以是4G空间内的任意虚拟地址。

PADDR(kva)：求kva对应的物理地址。kva必须是kseg0里的虚拟地址。

2.3 进程创建函数env_alloc

​ 函数int env_alloc(struct Env **new, u_int parent_id)的作用是创建并初始化一个以parent_id为父进程id的新进程。

​ 该进程仍然是ENV_FREE的状态，未加入调度链表，仅仅只是刚创建出来并完成了最基本的初始化。

2.4 内核态下创建可执行进程

​ 函数struct Env *env_create(const void *binary, size_t size, int priority)的作用是在内核态下读入ELF文件并依此创建一个指定优先级的就绪的进程。该函数先创建一个最原始的进程，设置其优先级及env_status，然后加载二进制镜像，最后将其加入调度队列的尾端。

​ 当异常发生时，当前进程会被挂起，然后硬件会帮我们执行以下操作：

​ 例如子进程首次被调度后到底应该从哪一条语句开始执行呢？为什么？

​ 应当是从child = syscall_exofork()处开始执行且正从syscall_exofork返回（msyscall的jr ra）0。子进程的env->env_tf源自于父进程调用syscall_exofork自陷时，在异常分发程序处SAVE_ALL的结果。后来子进程在schedule中被调度，从env_run处逐渐开始苏醒，经过env_pop_tf，最终在ret_from_exception恢复寄存器状态，并跳转到之前EPC寄存器中保留的地址。而该地址就是父进程自陷时4+PC寄存器的值，即汇编函数msyscall的jr ra。此后，子进程携带着值为0的v0寄存器返回syscall_exofork，最终child值为0。
